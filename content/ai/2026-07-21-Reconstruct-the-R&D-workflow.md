# 使用 AI 重构研发工作流初步设计

## 一、前期能力设计

重构的第一步就是拆解现有的研发流程

### 1、明确前端研发流程

产品需求
→ 阅读 PRD
→ 查看 Figma
→ 确认接口
→ 定位已有代码
→ 制定修改方案
→ 编写代码
→ 补测试
→ 本地验证
→ 提交 MR
→ Code Review
→ 修改问题
→ 发布

除此之外，还需要上下文工程能力，把零散的信息整理成 AI 可以理解的结构。

### 2、明确上下文结构
- 项目架构说明
- 目录职责
- 技术栈和版本
- 编码规范
- 组件使用规范
- 接口文档
- 业务术语
- 历史技术决策
- 常见错误
- 测试和发布方式
- 禁止修改的区域
  
例如，不要只告诉 AI：

``帮我开发用户列表页面``

而应该提供：

```
目标：
新增用户列表页，支持搜索和分页。

约束：
- 使用现有 Table 组件
- 请求必须经过 services/user.ts
- 不允许直接使用 axios
- 状态使用项目现有 hooks
- 需要补充 Playwright 用例

验收：
- 搜索条件刷新后保留
- 空状态符合设计系统
- 单元测试和类型检查通过
```

### 3、明确能力模块职责、能力

例如：

- 读取代码仓库
- 搜索 GitLab Issue
- 获取接口文档
- 读取 figma
- 执行 lint、typecheck、test
- 创建分支
- 生成 Commit
- 创建 Merge Request
- 汇总 Review 意见
- 分析 CI 失败原因

### 4、评估与质量控制能力

AI 工作流不能只看“代码生成得快不快”，还要判断它是否真的提高了研发效率以及质量是否达标。

比如记录指标：
```
1、需求交付周期：从开始开发到合并耗时
2、首次通过率：AI 第一次提交是否通过检查
3、人工修改率：AI 代码被人工重写的比例
4、Review 时间：人工审核花费时间
5、缺陷逃逸率：上线后出现问题的比例
6、AI 建议采纳率：AI 输出实际被采用多少
7、上下文缺失率：因信息不足导致失败的次数
```

质量检查：
```
1、代码规范检查
2、TypeScript 类型检查
3、单元测试
4、集成测试
5、E2E 测试
6、安全扫描
7、依赖检查
8、变更范围检查
9、人工 Review
```

## 二、完整能力地图

如果是想要搭建 AI 研发效能平台，通常需要以下能力组合：
| 能力方向  | 具体内容                 |
| ----- | -------------------- |
| 软件工程  | 架构、测试、CI/CD、代码质量     |
| 平台工程  | GitLab、流水线、权限、环境、工具链 |
| AI 工程 | 模型、Agent、工具调用、上下文管理  |
| 研发效能  | 流程分析、瓶颈识别、指标设计       |
| 产品能力  | 识别真实需求，控制平台复杂度       |
| 数据能力  | 收集过程数据，分析实际效果        |
| 安全治理  | 权限、代码泄露、数据隔离、审计      |
| 组织推动  | 规范落地、培训、反馈和迭代        |

作为技术，首先我会考虑的是：
1、研发流程拆解
2、上下文工程
3、Skills/Agent 编排
4、自动化验证和评估

## 三、做一个“前端需求到 MR”的半自动工作流

### 流程设计
输入需求
1、AI 分析需求
2、AI 搜索仓库
3、AI 生成实施计划
4、人确认方案
5、AI 修改代码
6、AI 执行验证
7、AI 自查 Diff
8、AI 生成 MR 描述
9、人最终 Review

### 初步建立项目知识
在仓库里建立一套最小上下文

比如：
```
.ai/
├── project-overview.md
├── architecture.md
├── coding-rules.md
├── testing-guide.md
├── business-glossary.md
├── common-mistakes.md
├── quality-gates.md
└── skills/
```

内容上：
```
禁止直接请求接口
禁止跨模块引用内部组件
新增页面必须接入权限系统
表格统一使用 AntDesign Table
表单统一使用 React Hook Form
不得修改自动生成的 API 类型
所有文案必须进入国际化文件
```

每次 AI 犯错，就把对应规则补充进去。这样 AI 工作流会随着实际使用持续进化。

### 设计一组 Skills
1、analyze-requirement
2、explore-repository
3、create-implementation-plan
4、implement-change
5、verify-change
6、review-diff

### 建立固定的人机检查点

不要让 Agent 从需求一直自动运行到合并。
至少要有三个检查点：
1、需求理解是否正确
2、实施计划是否合理
3、最终 Diff 是否可合并

流程类似：
AI 分析需求
↓
人确认

AI 生成方案
↓
人确认

AI 编码和验证
↓
人 Review 并合并
